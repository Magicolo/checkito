# Exhaustive Mode for `Any<[G; N]>` Generates Duplicate Values

## Summary

In exhaustive mode, `Any<[G; N]>` (a slice-style `any` with a fixed-size array of generators)
generates each generator's values in sequence.  However, if two generators in the array have
overlapping value ranges, the same value may be generated multiple times during exhaustive
iteration.  This reduces the effectiveness of exhaustive testing since some values are visited
more than once at the expense of other values not being visited at all.

## Affected Code

`checkito/src/any.rs` – `Generate for Any<[G; N]>` in exhaustive mode:

```rust
fn generate(&self, state: &mut State) -> Self::Shrink {
    Shrinker(
        state.any_uniform(self.as_ref()).map(|generator| generator.generate(state)),
    )
}
```

And `state.any_uniform` in exhaustive mode:

```rust
Mode::Exhaustive(index) => {
    generators.get(
        Self::any_exhaustive(index, generators.iter().map(G::cardinality))?
    )
}
```

## Example

```rust
let gen = any([0u8..=5, 3u8..=8]);
let values: Vec<u8> = gen.checks(|_| true).map(|r| r.into_item().unwrap()).collect();
// Expected (no duplicates):  [0, 1, 2, 3, 4, 5, 6, 7, 8]
// Actual (may have duplicates): [0, 1, 2, 3, 4, 5, 3, 4, 5, 6, 7, 8]
// because generator 0 produces 0..=5 and generator 1 produces 3..=8,
// and 3, 4, 5 appear in BOTH generators.
```

## Why This Is Mostly by Design

The `Any` combinator chooses from generators independently.  The library does not track
which values have been generated across generators.  When the ranges overlap, values in the
overlap are generated by both generators.

This is the same behavior as using `any` in random mode — both generators can produce the
same value.  The difference in exhaustive mode is that users might expect the exhaustive
iteration to cover the **union** of the value spaces without repetition.

## Impact

- **Correctness:** Tests are still valid — all values in both ranges are covered.  The issue
  is only the presence of duplicates (some values visited ≥ 2 times), which is not wrong but
  may be surprising.
- **Efficiency:** In the worst case, using `any([0u8..=10, 0u8..=10])` doubles the work since
  both generators produce the same values.

## Proposed Enhancement

This is a minor quality-of-life improvement, not a critical bug.  The documentation should
clarify:

1. `Any` in exhaustive mode iterates each generator's values in sequence, not the distinct
   union of all values.
2. If a true union (no duplicates) is needed, users should use ranges that don't overlap.

### Add Documentation Note

In `prelude.rs` for `any()` and in `any.rs`:

```rust
/// In exhaustive mode, [`Any`] iterates each generator's values independently.
/// If multiple generators can produce the same value, that value will appear
/// multiple times in the exhaustive sequence.  For a true union with no
/// duplicates, use non-overlapping generator ranges.
```

## No Code Change Required

This is primarily a documentation issue.  The current behavior is reasonable and by design.
If users need deduplication, they can use a `HashSet` or ensure non-overlapping ranges.
