# Incomplete TODO in lib.rs - Add Fuzzing Support

## Summary
There's a TODO comment in `state.rs` suggesting adding fuzzing support via `Fuzz(Box<dyn Iterator<Item = byte>>)` mode, but this feature is not implemented. This represents a significant missing capability for integration with fuzzing tools.

## Context
Property-based testing and fuzzing are complementary techniques:
- **Property testing**: Generates structured inputs based on generators
- **Fuzzing**: Mutates byte streams to find crashes/bugs
- **Hybrid**: Use fuzzer-generated byte streams as input to generators

The TODO suggests adding a `Fuzz` mode that would consume a byte iterator to drive generation, enabling integration with tools like cargo-fuzz, AFL, LibFuzzer, etc.

## The TODO Comment

**Location**: `checkito/src/state.rs:85-86`

```rust
// TODO: Can I use this for fuzzing? Add a Fuzz(Box<dyn Iterator<Item = byte>>)? Or
pub enum Mode {
    Random(Random),
    Exhaustive(u64),
}
```

**What's Missing**: A third variant:
```rust
pub enum Mode {
    Random(Random),
    Exhaustive(u64),
    Fuzz(Box<dyn Iterator<Item = u8>>),  // Missing!
}
```

## Current Limitations

### No Fuzzer Integration
Users cannot:
1. Use checkito generators with cargo-fuzz or AFL
2. Replay fuzzer-discovered inputs through generators
3. Use fuzzer corpuses for structured fuzzing
4. Combine coverage-guided fuzzing with property testing

### Workaround is Awkward
Current fuzzing requires:
```rust
// Manual deserialization from bytes
#[no_mangle]
pub extern "C" fn LLVMFuzzerTestOneInput(data: &[u8]) -> i32 {
    // Manually parse bytes into structured data
    // Can't use checkito generators!

    let value = i32::from_le_bytes([data[0], data[1], data[2], data[3]]);
```

## Proposed Implementation

### 1. Add Fuzz Mode
```rust
pub enum Mode {
    Random(Random),
    Exhaustive(u64),
    Fuzz(Box<dyn Iterator<Item = u8> + Send>),
}
```

### 2. Consume Bytes in Generators
```rust
impl State {
    fn integer<I: Integer>(&mut self, start: I, end: I) -> I {
        match &mut self.mode {
            Mode::Random(random) => { /* existing code */ },
            Mode::Exhaustive(index) => { /* existing code */ },
            Mode::Fuzz(bytes) => {
                // Consume bytes to generate value in range
                let num_bytes = std::mem::size_of::<I>();
                let mut buf = vec![0u8; num_bytes];
                for i in 0..num_bytes {
                    buf[i] = bytes.next().unwrap_or(0);
                }
                let value = I::from_le_bytes(&buf);
                // Scale to range [start, end]
                start + (value % (end - start + 1))
            }
        }
    }
}
```

### 3. Provide Helper for Fuzzer Integration
```rust
pub fn from_fuzz_input(bytes: &[u8]) -> State {
    State {
        mode: Mode::Fuzz(Box::new(bytes.iter().copied())),
        ..Default::default()
    }
}

// Usage with cargo-fuzz:
#[no_mangle]
pub extern "C" fn LLVMFuzzerTestOneInput(data: &[u8]) -> i32 {
    let mut state = checkito::from_fuzz_input(data);
    
    // Use checkito generators!
    let gen = (0..1000, letter().collect::<String>());
    let (num, text) = gen.generate(&mut state);
    
    test_property(num, &text);
    0
}
```

## Challenges

### 1. Byte Exhaustion
What happens when fuzzer input is too short?
```rust
// Fuzzer provides 10 bytes
// Generator needs 100 bytes
// Solution: Return default values or repeat last byte
```

### 2. Determinism
Byte stream must produce deterministic output:
```rust
// Same bytes must always produce same value
// Important for shrinking and reproduction
```

### 3. Efficiency
Converting bytes to structured data should be fast:
```rust
// No complex deserialization
// Simple arithmetic operations
```

### 4. Bias
Naive byte consumption can create bias:
```rust
// Random bytes uniformly distributed
// But range [0..100] might not be uniform
// Need careful mapping strategy
```
